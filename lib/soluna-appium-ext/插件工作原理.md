# soluna-ext 与 Appium 的协作原理

本文解释插件在 Appium 中的接入方式、请求处理链路以及当前关键实现（含日志会话能力）。

## 1. 插件如何被 Appium 识别并加载

`package.json` 中声明了：

- `appium.pluginName`: `soluna-ext`
- `appium.mainClass`: `SolunaExtPlugin`

当使用 `appium --use-plugins=soluna-ext` 启动时，Appium 会加载 `lib/plugin.ts` 中的 `SolunaExtPlugin`，并调用 `updateServer(...)`。

## 2. `updateServer` 是协作入口

`SolunaExtPlugin.updateServer(expressApp, httpServer)` 会做两件事：

1) 启动前检查：`runPreflightChecks()`
2) 挂载插件路由到 Appium 的 Express 实例

当前路由总览：

- `GET /soluna/device`：按 UDID 查询单设备
- `GET /soluna/devices`：查询当前所有连接设备
- `POST /soluna/command`：执行 `adb/go-ios/ios` 命令
- `POST /soluna/logs/sessions`：创建日志会话
- `GET /soluna/logs/sessions/:sessionId`：按 cursor 增量拉日志
- `DELETE /soluna/logs/sessions/:sessionId`：停止并删除日志会话

这意味着插件并不是独立 HTTP 服务，而是扩展了 Appium 自身服务。

## 3. 启动前检查（preflight）如何阻断启动

实现位于 `lib/cli/preflight.ts`：

- 按平台选择命令探测器：
  - Windows: `where`
  - macOS/Linux: `which`
- 检查必需工具：
  - `adb`
  - `go-ios` 或 `ios`
- 缺失时：输出错误并抛异常

异常发生在 `updateServer` 阶段，Appium 会将其视为插件初始化失败，从而阻止“带该插件”的可用启动。

## 4. 设备查询能力的处理链路

### 4.1 `GET /soluna/device?udid=...`

1. `handleGetDeviceInfo` 校验参数
2. `lookupDeviceByUdid` 聚合查询：
   - Android：`adb devices` + `adb -s <udid> shell getprop`
   - iOS：`go-ios|ios list --details`
3. 映射成统一模型 `UnifiedDeviceInfo`
4. 返回 Appium 风格响应：
   - 200: `{value:{exists:true,device}}`
   - 404: `{value:{exists:false,message}}`
   - 400: `{value:{error:"invalid_argument",...}}`

### 4.2 `GET /soluna/devices`

1. `handleListDevices`
2. `listConnectedDevices` 并发调用：
   - `listAndroidDevices`
   - `listIosDevices`
3. 聚合返回：
   - `{value:{count,devices}}`

## 5. 命令执行能力的处理链路

`POST /soluna/command`：

1. `handleExecuteCommand` 做请求校验
2. `executeSupportedCommand` 调用子进程执行
3. 执行层保障：
   - 超时终止（`SIGTERM` -> `SIGKILL`）
   - 输出上限截断（避免超大响应）
   - `go-ios` / `ios` 别名回退
4. HTTP 语义：
   - 200：命令退出码为 0
   - 422：命令执行成功但退出码非 0
   - 500：执行层异常

## 6. 统一日志会话能力（核心新增）

实现核心在 `lib/services/log-session-manager.ts`。

### 6.1 会话模型

创建会话后，插件维护：

- `sessionId`
- 设备信息：`udid`、`platform`
- 运行状态：`running/stopped/failed`
- 游标信息：`minSeq`、`nextSeq`、`droppedCount`
- 限制参数：`ttlMs`、`maxBufferEntries`、`maxSessionBytes`

并强制“同一 `udid` 仅允许一个运行中会话”。

### 6.2 平台命令映射

- Android：`adb -s <udid> logcat -v threadtime`
- iOS：`ios|go-ios syslog --udid <udid>`

### 6.3 存储策略（防内存爆）

采用“落盘为主 + 小内存缓存”：

- 每条日志写入临时 JSONL 文件（`tmp/soluna-appium-ext/log-sessions/*.jsonl`）
- 内存中维护 ring buffer（最近 N 条）
- 文件超阈值会裁剪最旧日志并维护 `droppedCount`
- `cursor` 过旧会自动调整并通过 `cursorAdjusted` 反馈给调用方

### 6.4 生命周期管理

- 支持主动删除会话（DELETE）
- 定时清理 TTL 过期会话
- 进程退出前通过 `beforeExit` 触发 `dispose()`，停止子进程并清理临时文件

### 6.5 curl 调用示例（日志接口）

```bash
APPIUM_URL="http://127.0.0.1:4723"
UDID="<YOUR_UDID>"
```

创建会话：

```bash
curl -sS -X POST "$APPIUM_URL/soluna/logs/sessions" \
  -H "Content-Type: application/json" \
  -d "{
    \"udid\":\"$UDID\",
    \"maxBufferEntries\":1000,
    \"maxSessionBytes\":104857600,
    \"ttlMs\":600000
  }"
```

读取日志（首次）：

```bash
curl -sS "$APPIUM_URL/soluna/logs/sessions/<SESSION_ID>?cursor=0&limit=200"
```

读取日志（增量）：

```bash
curl -sS "$APPIUM_URL/soluna/logs/sessions/<SESSION_ID>?cursor=<NEXT_CURSOR>&limit=200"
```

删除会话：

```bash
curl -sS -X DELETE "$APPIUM_URL/soluna/logs/sessions/<SESSION_ID>"
```

日志过多或长时间未拉时，若你只想从“最新位置”继续：

```bash
curl -sS "$APPIUM_URL/soluna/logs/sessions/<SESSION_ID>?cursor=999999999&limit=1"
```

然后使用响应中的 `nextCursor` 持续增量拉取即可。
若不保留旧上下文，也可以直接删除旧会话后新建会话。

## 7. 与 Appium 日志系统的协作

命令路由（`lib/http/command-route.ts`）日志策略：

- `info`：请求摘要与结果摘要
- `warn/error`：校验失败与执行异常
- `debug`：`stdout/stderr` 预览输出（做长度截断，避免日志污染）

## 8. Python client 在协作中的位置

根目录 `soluna_client.py` 是唯一 Python 客户端入口，已覆盖全部 HTTP 接口：

- 设备查询：`get_device` / `get_device_info` / `list_devices`
- 命令执行：`execute_command`
- 日志会话：`create_log_session` / `read_log_session` / `delete_log_session` / `log_session(...)`

## 9. 为什么保持 route / service / cli 分层

- `http/*`：HTTP 参数与响应语义
- `services/*`：业务编排与平台差异
- `cli/*`：外部命令执行能力

好处：

1. 更易测试（依赖可注入）
2. 更易扩展（新增路由/平台影响面小）
3. 更符合 Appium 插件“增强能力”模式

## 10. 一句话总结

插件通过 `updateServer` 挂入 Appium 的同一 HTTP 服务，在启动时做依赖守卫，在运行时提供设备查询、命令执行和跨平台日志会话能力，并保持统一响应与可观测性语义。
